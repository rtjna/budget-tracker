## Part 1 — Cold codebase audit

### HIGH

**H1 — Unlisted currencies silently converted 1:1 to GBP in all dashboards**
`backend/app/stats.py:46` (`to_gbp` → `GBP_RATES.get(currency, Decimal("1"))`), rates table at `stats.py:19-26`.
`to_gbp` falls back to a rate of **1.0** for any currency not in the six-entry `GBP_RATES` table, and the Revolut importer (`importers/revolut.py:72`) creates a separate account for *any* currency the export contains (`Revolut {currency}`). So spending in SEK, NOK, INR, AUD, CAD, PLN, etc. is summed into GBP totals at 1:1 with no warning.
Failure scenario: a user imports a Revolut SEK statement (SEK ≈ £0.075). Every krona is counted as a pound, overstating that month's spending, the category ranking, and recurring-payment detection by ~13×. The "approximate rates" caveat covers rounding, not a silent order-of-magnitude error for a first-class importer.

### MEDIUM

**M2 — Splitwise items in unsupported currencies are dropped permanently, not just skipped**
`backend/app/splitwise.py:121-123, 223-224, 257-259`.
`_to_gbp` returns `None` for a currency missing from `GBP_RATES`; both the settle-up and correction branches then `continue` *before* the row is added or fingerprinted. Because nothing is written to `LlmMerchantCache`/`Transaction`, the expense is silently re-fetched and re-dropped on every sync and never appears.
Failure scenario: a shared expense recorded in Splitwise in DKK never shows up in the app; the user has no signal it was skipped, so their Splitwise-adjusted spending is understated indefinitely.

**M3 — Duplicate category or rule creation returns an unhandled 500**
`backend/app/main.py:555-560` (`create_category`) and `main.py:628-640` (`create_rule`).
`Category` has `UniqueConstraint("name","parent_id")` and `Rule` has `UniqueConstraint("match","pattern")` (`models.py:33,98`), but neither POST handler catches `IntegrityError` around `db.commit()`. A duplicate name/pattern raises an uncaught exception surfaced as HTTP 500 with a raw stack, rather than a 409/422. (Contrast `review_assign`, which does pre-check for an existing rule.)
Failure scenario: user re-adds a category named "Groceries" or a rule that already exists → opaque 500.

**M4 — `to_gbp`'s 1:1 fallback also corrupts recurring detection and category drill-downs**
`backend/app/stats.py:325,331,345` (recurring) and `stats.py:273` (`category_merchants`).
Same root cause as H1 but distinct blast radius: `recurring()` compares amounts and computes `monthly_equivalent` across mixed currencies all treated as GBP, so a foreign-currency subscription's "typical amount", price-change flag, and monthly-equivalent are wrong, and its variance check (`(max-min)/typical > 0.6`) can wrongly include/exclude it.

**M5 — Ordinal-based fingerprint dedup can silently drop a genuinely new identical transaction**
`backend/app/importing.py:71-73` and `importing.py:146-155`.
Fingerprints are `sha256(account|date|amount|description|ordinal)`, where `ordinal` is the occurrence index *within the current file's* (account, date, desc, amount) group. If a later, non-overlapping import contains one identical (date, desc, amount) row that also existed earlier, it is assigned `ordinal=0` and collides with the earlier row's fingerprint, so it is counted as a duplicate and never stored.
Failure scenario: two identical £3.00 "COFFEE" charges on the same day were imported from statement A; a corrected re-export or a same-day third identical charge arriving alone gets `ordinal 0/1` and is dropped as a dup, under-counting spend. Narrow (needs identical date+desc+amount across separate imports) but real, and there is no test for it.

### LOW

**L6 — Monzo token file without `obtained_at` crashes token refresh**
`backend/app/monzo.py:121`: `datetime.fromisoformat(tokens.get("obtained_at"))` raises `TypeError` (500) if an older/hand-edited `monzo_tokens.json` lacks the key, instead of treating the session as expired.

**L7 — OAuth `state` set is in-memory only**
`backend/app/monzo.py:38` (`_pending_states`). A server restart between `/connect` and `/callback` (or any second worker) discards pending states, so the callback fails with "Unknown OAuth state". Acceptable for single-process, but brittle.

**L8 — Frontend chart math breaks on empty data**
`frontend/src/Dashboard.tsx:568` and `:754`: `Math.max(...months.map(...))` returns `-Infinity` when `months` is empty, producing `NaN` geometry. Also `frontend/src/App.tsx:88-94`: `new Intl.NumberFormat(..., {currency})` throws `RangeError` if an account's `currency` is not a valid ISO-4217 code.

**L9 — Splitwise category map points at non-existent categories**
`backend/app/splitwise.py:44,70`: `CATEGORY_MAP` maps "Food and drink" → "Food" and "Education" → "Education", neither of which is in `SEED_CATEGORIES` (`categorize.py:8-27`). `categories.get(...)` returns `None`, so these silently land uncategorized rather than in an intended bucket — misleading dead mapping.

**L10 — Minor consistency/robustness nits**
- `frontend/src/App.tsx:556`: delete-confirmation dialog hardcodes `money(t.amount, 'GBP')` while the table row (`:548`) uses the account's real currency — a non-GBP amount is shown with a £ sign in the prompt.
- `backend/app/secrets_env.py:53`: `os.environ.get(name) or _from_keychain(name)` treats an empty-string env var as unset and silently falls through to the Keychain.
- `backend/app/db.py:23-26`: SQLite engine sets no `timeout` in `connect_args`; concurrent writes (e.g. a Monzo sync overlapping a manual add, both running in FastAPI's threadpool with `check_same_thread=False`) rely on pysqlite's default 5s busy timeout and can surface "database is locked".
- `backend/app/main.py:172-173`: `data = await file.read()` loads the entire upload into memory with no size cap (loopback-only, so low risk).

### Overall assessment

This is a carefully engineered, security-conscious codebase for its threat model. The CSRF-via-custom-header + `TrustedHostMiddleware` design is sound: state-changing requests without `X-Budget-App: 1` are rejected, cross-origin requests carrying it force a preflight that CORS grants only to the Vite origin, the DNS-rebinding host filter is correct, and the OAuth callback's GET exemption is protected by `state`+`code`. I found **no CSRF/host/CORS bypass**. Secrets handling (Keychain → env → gitignored file), Monzo/Splitwise idempotency via stable external IDs, transfer-pair exclusion to avoid double-counting, sign conventions across importers, and the self-validating scrambled-font PDF parsers (arithmetic reconciliation with loud failure) are all genuinely solid, and test coverage is broad (119 tests, all passing).

The real risk concentration is **data integrity around multi-currency**: the silent 1:1 fallback in `stats.to_gbp` (H1/M4) and the permanent silent drop of unsupported-currency Splitwise items (M2) can materially misstate the app's core output for anyone banking in foreign currencies — which, given Revolut is a headline importer, is a plausible real user. The remaining items are ordinary robustness gaps (unhandled uniqueness violations, empty-data edge cases, a couple of 500-on-bad-input paths). Nothing suggests double-counting of transfers or broken dedup in the common path; the dedup weakness (M5) is narrow. Highest-value fixes: make unknown currencies fail loudly rather than convert 1:1, and surface skipped Splitwise items.

## Part 2 — UI/UX & visual-design audit

Two visual systems coexist: **Dashboard.css/Coverage.css** use a proper CSS-variable, theme-aware token system scoped to `.viz-root`; **App.css/index.css** (the whole Transactions/Review shell plus the app chrome) use ad-hoc hardcoded `#888x` alphas and a stray Vite-purple accent. Most findings below trace back to that split.

### CRITICAL

**C1. All `data-tip` tooltips are invisible to screen readers, and half of them are unreachable by keyboard.** `index.css:32-56`. The tooltip is a pure CSS `::after` with `content: attr(data-tip)` — it never enters the accessibility tree, so assistive tech gets nothing. Worse, `data-tip` is attached to many *non-interactive* elements (`.account-card` App.tsx:345, `.stat-tile` Dashboard.tsx:513, `.source-badge` App.tsx:522/540, `.filepick` App.tsx:313), which cannot receive focus, so the `:focus-visible` reveal (index.css:59) never fires for them either. Load-bearing information lives only in these bubbles: what `rule`/`model`/`llm` badges mean, what "⇄ transfer" means, the "Net invested" definition. Keyboard and SR users lose all of it. Fix: for informational tips on static elements, render real text/`<abbr title>`/an `aria-label`ed element, or make them `tabindex=0` with `role` + an accessible name; don't rely on the CSS pseudo-element as the only channel.

**C2. "Choose files" is keyboard-inaccessible.** App.tsx:311-323. The trigger is a `<label className="filepick">` wrapping a `hidden` `<input type=file>`. A `<label>` isn't in the tab order and a `hidden` input isn't focusable, so there is no keyboard path to the primary import action. It also reads as underlined body text (App.css:25-29), not a control. Fix: use a real `<button>` that calls `input.click()`, keep the input visually hidden but focusable (`sr-only`, not `hidden`), or make the label focusable and Enter/Space-activatable.

**C3. Interactive charts and bar rows have no visible focus indicator.** No `:focus`/`outline` rule exists anywhere except the data-tip (confirmed by grep). The SVG `<g tabIndex={0}>` bars (Dashboard.tsx:438-445, 629-638, 774-783) and the `role="button" tabIndex={0}` category rows (Dashboard.tsx:354-358) are keyboard-focusable but paint no ring — browsers frequently render no default outline on SVG `<g>`, so keyboard users can't tell what's focused. The category rows also handle Enter but not Space (native buttons activate on both). Additionally each focusable `<g>` has no `aria-label`, so focusing a bar announces nothing. Fix: add a visible `:focus-visible` outline for `.bar-row` and the chart `<g>`, handle Space, and give each bar an `aria-label` (month + total).

### HIGH

**H1. App shell is not theme-aware; it hardcodes light-biased grays and a foreign accent.** All of App.css uses `#8883`–`#888a` alpha-gray borders and literal colors instead of the `--gridline`/`--baseline`/`--text-*` tokens the dashboard uses (which only exist inside `.viz-root`). Because the Transactions/Review trees live outside `.viz-root`, they can't use the tokens at all. Result: two different border treatments and text-contrast behaviors across tabs. Fix: promote the `viz-root` token block to `:root` and consume it everywhere.

**H2. Spending amounts have no color rule — income-vs-spending is encoded by the *presence* of green alone.** App.tsx:547 sets `className={num ${t.amount<0?'out':'in'}}` but `.out` does not exist (confirmed by grep); only `.tx-table .in { color:#2a9d4a }` (App.css:100). So income is green and spending is default text — the sign is the only other cue. The green `#2a9d4a` is hardcoded (not tokenized) and on the light body it computes ~3.9:1 for 14.7px text, under the 4.5:1 AA threshold. Fix: tokenize both in/out colors, ensure AA in both themes, and don't rely on one-sided color.

**H3. `--text-muted: #898781` fails contrast in light mode across the dashboard.** Dashboard.css:6 on `--surface-1:#fcfcfb` yields ~3.3:1. It's used for `axis-text` (10.5px, Dashboard.css:217), `dash-note`, `stat` sub-labels, muted table cells, `coverage-kind` — all small text, all below AA. Several are compounded by `opacity` (`.source-badge` .7 App.css:154, `.review-meta` .7, `.coverage` .7). Fix: darken the light-mode muted token to ≥4.5:1 and stop stacking opacity on already-muted text.

**H4. The SVG hover tooltip is pinned to the top-right corner and occludes the most recent months.** `.viz-tooltip { position:absolute; top:8px; right:8px; min-width:12rem }` (Dashboard.css:226-239); markup Dashboard.tsx:671, 815, 465. It's fixed regardless of which bar you hover, so it permanently covers the right edge of the plot — exactly where the latest month (the one users care about most) sits. On a narrow viewport a 12rem box covers most of the chart. Fix: position the tooltip opposite the hovered bar (or above it), or shrink/inline it so it never overlaps data.

**H5. The Transactions filter bar does not wrap and overflows on narrow screens.** `.filters { display:flex; gap:.75rem }` with **no `flex-wrap`** (App.css:61-65) contains a search input plus two selects, a checkbox, and four action buttons ("Detect transfers", "Sync Splitwise", "Sync Monzo", "+ Add transaction"). Below ~800px these eight controls either overflow horizontally or crush the `flex:1` search box to nothing. This is the app's worst responsive break. Fix: `flex-wrap: wrap`, and consider splitting the sync/import actions into their own toolbar row.

**H6. Loading and empty states are conflated on the Dashboard, and fetch errors hang forever.** Dashboard.tsx:110-112 returns "No data yet — import some transactions first." whenever `overview` is null — which is also the initial pre-fetch state, so a user with data sees the "no data" message flash on every load. Coverage.tsx:42-43 does it correctly (separate "Loading…" vs "No data yet."), so the two tabs are inconsistent. None of the dashboard/coverage fetches have `.catch` (Dashboard.tsx:76-95, Coverage.tsx:36-40), so a backend error leaves the tab stuck on "No data yet"/"Loading…" with no error surfaced. `loadTxs`/`loadReview`/`loadStatic` (App.tsx:203-231) likewise swallow failures, silently showing an empty table. Fix: distinguish loading/empty/error, and render an error state on fetch rejection.

### MEDIUM

**M1. Three different tooltip mechanisms.** Dark CSS `data-tip` bubbles (index.css), the surface-colored pinned `.viz-tooltip` (Dashboard.css:226), and native `title=` on BarList rows (Dashboard.tsx:872) and every Coverage cell (Coverage.tsx:99). Three look-and-feels, three behaviors, three accessibility profiles. Pick one system.

**M2. `data-tip` bubbles have no viewport/overflow collision handling.** index.css:33-40 always renders the bubble below the element, centered with `left:50%; translateX(-50%)`, `max-width:260px`. Right-edge triggers (last tab "Coverage" App.tsx:391, the rightmost filter buttons, `.stat-tile` at the grid's right column) push a 260px bubble off the right edge on sub-900px viewports; bottom-of-page triggers (the pager buttons App.tsx:571-582, the long TrainModel paragraph App.tsx:785) render below the viewport. Pure CSS can't flip. Fix: either a JS-positioned tooltip with edge detection, or at minimum constrain/anchor edge cases.

**M3. Two different "primary blue" accents.** The shell uses Vite's default `#646cff` purple (dropzone drag, `.filepick`, active tab — App.css:17,18,26,121,122) while the dashboard's primary series is `#2a78d6` (Dashboard.css:11). They read as two brands. The active-tab indicator is also *color-only* (`.tabs button.active` sets only border+text color, App.css:120-123) with no underline/weight and no `aria-current`/`aria-selected` — weak affordance and a color-alone state. Fix: one accent token; add a non-color active-tab cue and `aria-current`.

**M4. `--series-4` green is identical in both themes and collides with two neighboring greens.** Dashboard.css:35 and :34 both `--series-4:#008300`; every other series was brightened for dark mode, so this dark-saturated green sits poorly on the `#1a1a19` dark surface. It's also one of three green/teal hues (`--series-2:#1baf7a`, `--series-3:#eda100`, `--series-4:#008300`) that are hard to distinguish as adjacent stacked segments. Fix: give series-4 a dark-mode variant and re-space the categorical hues.

**M5. Same colors mean different things on adjacent dashboard charts.** In "Income vs spending" (Dashboard.tsx:753-759), spending = series-1 (blue), income = series-2 (green). In the stacked "Spending by month" directly above, series-1/series-2 are the #1 and #2 *categories*. So blue means "top category" in one card and "spending total" in the next. Fix: give income/spending their own dedicated semantic tokens, distinct from the categorical ramp.

**M6. Negative category totals silently vanish from and distort the stacked chart.** The API returns negative `by_category` values (confirmed: `"6":-61.66` in 2025-08). In `StackedColumns.stackFor` (Dashboard.tsx:578-585) negative slot values are dropped by `.filter(p=>p.value>0)` yet still subtracted inside the `other` computation, so a refund-heavy category disappears from its own segment and inflates "Smaller categories." The chart table (Dashboard.tsx:717) can then show figures that don't reconcile. Fix: decide on a refund/netting policy and represent it consistently in chart, tooltip, and table.

**M7. Coverage heatmap has no legend for its intensity buckets.** Coverage.tsx thresholds (1–4 / 5–14 / 15–39 / 40+, from BUCKETS at :19-24) are never explained; the user sees a 4-step blue ramp with no key. The printed cell numbers make color redundant (good), but a small legend would make the ramp legible. Also, note the ramp direction inverts between themes (light: high=dark; dark: high=light) — correct for the differing surfaces, but a legend should reflect whichever theme is active.

### LOW

**L1. Inconsistent spelling convention.** American `-ize` ("Categorize"/"categorized", App.tsx:386,593) mixed with British `-ise` ("authorising"/"Authorise", App.tsx:171,486) despite a uniform `en-GB` locale. Pick one.

**L2. Hardcoded currency in user-facing copy.** The delete `confirm()` always formats as GBP (App.tsx:556 `money(t.amount,'GBP')`) even for EUR/USD accounts; AddTransaction hardcodes "Amount £" (App.tsx:694) and the label "Money in" (App.tsx:687) diverges from "Income" used everywhere else.

**L3. `daysAgo(null)` renders awkward copy.** App.tsx:98 returns "no data", producing "activity no data" on account cards (App.tsx:360). Special-case the phrasing.

**L4. Heading hierarchy skips a level.** `<h1>` (App.tsx:295) jumps straight to `<h3>` in chart cards (Dashboard.css/Dashboard.tsx:531); tab panels have no `<h2>`. Tabs are plain `<button>`s in a `<nav>` with no `role="tablist"/tab"` semantics. Minor semantic/SR-navigation cost.

**L5. Truncated names with no recovery.** Dashboard mini-table merchant cells truncate at `max-width:18rem` (Dashboard.css:353-359) with no `title`, unlike the BarList rows which do have one — long merchant names become unreadable. `transferMsg` success and error share `.review-intro` styling (App.tsx:481-491) so "Monzo sync failed" looks identical to a success message.

**L6. Scattered radii/one-offs.** Border-radius ranges 5/6/8/10/12/999px with no scale; the lone `5px` on `.cat-cell select` (App.css:143) is an outlier. `.tx-table` (App.css:81) has no `overflow-x` wrapper, so it can overflow on phones unlike the Coverage grid which correctly wraps in `.coverage-scroll`.

### Quick wins (highest value / lowest effort)
1. Add `flex-wrap:wrap` to `.filters` (H5).
2. Add a `.out` color + tokenize `.in` (H2).
3. Add a global `:focus-visible` outline and Space-key handling (C3).
4. Split Dashboard loading vs empty, add `.catch` error states (H6).
5. Make "choose files" a real focusable button (C2).
